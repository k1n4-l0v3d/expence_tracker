# Rate Limiting + Captcha — Design Spec

**Date:** 2026-04-16  
**Project:** Expense Tracker (Flask)  
**Status:** Approved

---

## Цель

Защитить маршруты `/login` и `/register` от брутфорса и автоматических ботов.

---

## Три слоя защиты

### 1. Rate Limiting по IP

- Хранилище: словарь в памяти `Dict[str, {"count": int, "blocked_until": datetime}]`
- Порог: **10 неудачных попыток**
- Блокировка: **5 минут**
- Применяется к: `/login` (неверный пароль) и `/register` (любая ошибка валидации)
- При перезапуске сервера счётчики сбрасываются — приемлемо для текущего масштаба

**Класс `RateLimiter` в `app.py`:**
```
методы:
  is_blocked(ip) -> bool
  record_failure(ip)
  reset(ip)          # вызывается при успешном входе
```

### 2. Математическая капча

- Сервер генерирует пример: `a OP b`, где OP ∈ {+, -, ×}, значения 1–20
- Правильный ответ сохраняется в `session["captcha_answer"]`
- При каждом GET-запросе — новый пример
- Пользователь вводит ответ в числовое поле формы
- Проверка на POST: `int(request.form["captcha"]) == session["captcha_answer"]`
- При неверном ответе — перегенерировать пример, показать ошибку, **не** засчитывать как неудачную попытку авторизации

### 3. Honeypot

- Скрытое поле `<input type="text" name="website" tabindex="-1" autocomplete="off" style="display:none">`
- На POST: если `request.form.get("website")` не пустое — молча вернуть форму (не сообщать боту об обнаружении)
- Не засчитывается как неудачная попытка авторизации

---

## Порядок проверок на POST

```
1. Honeypot заполнен?        → вернуть форму без сообщения об ошибке
2. IP заблокирован?          → flash("Слишком много попыток. Подождите N мин."), вернуть форму
3. Капча верна?              → нет: flash("Неверный ответ на капчу"), перегенерировать, вернуть форму
4. Основная логика (auth)    → при неудаче: rate_limiter.record_failure(ip)
                             → при успехе: rate_limiter.reset(ip)
```

## Что не входит в скоуп

- Постоянное хранение счётчиков (Redis, БД) — нет необходимости для текущего масштаба
- reCAPTCHA / hCaptcha — требуют внешних API-ключей
- Блокировка по username — избыточно при наличии блокировки по IP
- Rate limiting для других маршрутов (API, смена пароля) — отдельная задача
